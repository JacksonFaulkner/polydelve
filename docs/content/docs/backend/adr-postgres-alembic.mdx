## Status

Accepted — June 2026

---

## Context

Polydelve's dev MVP used **DuckDB** locally and **MotherDuck** (DuckDB-as-a-service) in production. This was a deliberate fast-start choice:

- Zero infrastructure setup — a `.duckdb` file is the entire database
- MotherDuck gave us a cloud target with no ops work
- DuckDB's analytical SQL (window functions, `list_contains`, `array_agg`) made package/CVE aggregations easy to write
- Schema was managed via `init_db()` in `features/db.py` — a single function that ran `CREATE TABLE IF NOT EXISTS` plus a pile of `ALTER TABLE` calls wrapped in `_safe()` to handle already-applied migrations

This worked well for solo development. It stopped working the moment we prepared for real users.

### Problems with the DuckDB approach at launch

**Concurrency.** DuckDB is a single-writer database. The app used a global `threading.Lock()` to serialize all writes. Under any real concurrent load — multiple users buying contracts simultaneously — requests queue behind the lock. A slow write blocks everything.

**Schema drift risk.** The `_safe()` migration pattern had no version tracking. Every app startup re-ran every `ALTER TABLE`. There was no way to know which statements had been applied to which environment. Adding a column meant hoping `_safe()` swallowed the "already exists" error cleanly.

**Crash safety.** DuckDB's WAL gives some protection, but a mid-write crash with a shared connection could corrupt the file. No automated backups in the MotherDuck free tier.

**Operational mismatch.** The rest of the stack (ECS, CloudFront, S3) is standard AWS. DuckDB/MotherDuck is a sharp outlier with no native AWS integration, no IAM auth, and no RDS-style automated failover.

---

## Decision

Migrate the transactional database to **PostgreSQL 16** (RDS `db.t4g.micro` in production, Docker locally) and manage schema with **Alembic**.

### Why Postgres

- Multi-writer, ACID, handles concurrent users without application-level locking
- RDS handles automated backups, failover, and parameter tuning
- `db.t4g.micro` (~$13/mo) is predictable cost — no autoscaling surprises
- Standard SQL; every DuckDB query translates with minor changes (`?` → `%s`, `list_contains` → `= ANY(...)`, `FLOAT[]` → `FLOAT[]`)
- Native AWS integration: VPC, IAM auth, Secrets Manager

### Why Alembic

- Version-controlled migrations with a tracked `alembic_version` table — each environment knows exactly what has been applied
- `upgrade` / `downgrade` per revision — safe rollback
- Replaces the entire `_safe()` hack with explicit, auditable SQL files
- Works with psycopg2 today; compatible with SQLModel/SQLAlchemy ORM if we add it later

### Why not SQLModel now

SQLModel (single class = Pydantic model + SQLAlchemy ORM) was evaluated and deferred. Migrating raw SQL queries to ORM calls is a larger scope than the DB swap itself. Post-launch refactor.

## Pydantic as the stability layer

The most important architectural decision that made this migration low-risk was one already in place: **all data crossing the API boundary was typed as Pydantic models** (`User`, `ContractDetail`, `PackageRow`, `EpssHistoryRow`, etc.) before this migration started.

Because routes mapped DB rows → Pydantic models → JSON responses, the migration reduced to:

1. Swap the connection layer (`duckdb.connect` → `psycopg2.connect`)
2. Translate SQL syntax (minimal changes)
3. Verify model fields still match DB columns

The Pydantic models were **completely unchanged** across the migration. They served as a contract between the DB layer and the API layer that neither side could break silently.

To enforce this going forward, `tests/test_schema_drift.py` introspects the live Postgres schema via `information_schema.columns` and asserts that every Pydantic DB-row model field exists as a column. Model fields added without a corresponding migration fail the test immediately.

## Migration path

```
DuckDB (local file)          →   Postgres 16 (Docker local / RDS prod)
MotherDuck (cloud DuckDB)    →   RDS db.t4g.micro
init_db() + _safe() hacks    →   Alembic versioned migrations
? placeholders               →   %s placeholders (psycopg2)
list_contains(col, val)      →   val = ANY(col)
array_agg(DISTINCT ...)      →   same (Postgres native)
::FLOAT[3072] cast           →   FLOAT[] (no size param)
datediff('day', a, b)        →   b - a (interval arithmetic)
```

## Consequences

**Positive**
- Concurrent writes work without application-level locking
- Schema state is auditable and reproducible across environments
- Pydantic drift test catches schema/model divergence at CI time
- Production cost is bounded and predictable (~$15/mo total DB cost)

**Negative**
- Local dev requires Docker running (vs. just a file)
- DuckDB's analytical query sugar (`list_cosine_similarity`, `list_contains`) needs to be rewritten or replaced with pgvector / Postgres array operators
- `news_repository.py` uses `list_cosine_similarity` for semantic dedup — needs `pgvector` or a Python-side cosine similarity fallback before that path works on Postgres
